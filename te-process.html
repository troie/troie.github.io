Test Engineer 工作流程規劃

需求確認

  • 項目說明：透過 PM 瞭解項目開發目的以及想解決的問題。
  • 頁面確認：透過 designer 或 PM 瞭解頁面規劃設計及功能細節。
  • 建立環境：項目會運行在哪些裝置，如 desktop, Mobile, Pad...建置適合的測試環境
  • 驗證瀏覽器：項目需測試那些瀏覽器。
  • 測試時程：測試起始及結束時間確認。
  • Test Cases：建立測試文件。
  • 測試資料：準備測試所需數據。

測試分為六個部分：

  1. 功能測試
  2. 性能測試(包括負載/壓力測試)
  3. 用戶界面測試
  4. 兼容性測試
  5. 安全測試
  6. 接口測試

功能測試

  • 連結測試：連結測試可分為三個部分。首先，測試所有連結是否按指示的那樣確實連結到了該連結的頁面;其次，測試連結的頁面是否存在；最後，確認 Website 沒有孤立頁面，所謂孤立頁面是指沒有連結指向該頁面，只有知道正確的URL地址才能訪問。

    測試工具：WebPageTest or Google Search Console

  • 表單測試：當用戶使用表單進行註冊，登錄，資料填寫等操作時，我們必須測試資料的完整性，以確認伺服器接收資料的正確性。例如：用戶填寫的出生日期與職業是否正確，填寫的省份與所在城市是否錯誤。如果使用了默認值，還要檢驗默認值的正確性。如果表單只能接受指定的某些值，也要進行測試。例如：只能接受某些符號，測試時可以跳過這些符號，看系統是否會出錯。
  • cookies 測試：Cookies 通常用來存儲用戶資料和用戶在某應用系統的操作記錄，當一個用戶使用Cookies 使用了某一個功能時，網頁伺服器將發送關於用戶的訊息，將該訊息以Cookies的形式存儲在客戶端電腦上，這可用來創建動態和客製化頁面或存儲登入訊息。

    測試工具：Chrome DevTools : Ctrl+Shift+I (Windows) or Cmd+Opt+I (Mac)

  • 資料庫測試：在使用資料庫的Web應用系統中，一般情況可能發生兩種錯誤，分別是數據一致性錯誤和輸出錯誤。數據一致性錯誤主要是由於用戶提交的表單信息不正確而造成的，而輸出錯誤主要是由於網絡速度或程序設計問題等引起的，針對這兩種情況，可分別進行測試。
  • 特定功能需求：下單、更改訂單、取消訂單、核對訂單狀態、在商品發送之前更改送貨訊息、線上支付等等。這些用戶使用網站的原因，一定要確認網站能提供正確的服務。

性能測試(包括負載/壓力測試)

用戶界面測試

兼容性測試

  • 測試平台：不同的操作系統，常見的有Windows，Mac OS，Linux等。
  • 測試瀏覽器：框架和層次結構在不同的瀏覽器中會有不同的呈現方式，甚至無法正常顯示。不同的瀏覽器對安全性和 Java 的設置也不一樣。
  • 分辨率測試：頁面版型在 640x400、600x800或1024x768 的分辨率模式下是否顯示正常？字體是否太小無法瀏覽？或太大？文本和圖片是否對齊？
  • 連接速度：測試頁面下載速度。用戶在下載文章或頁面時，可能會因等待時間過長，失去耐心等待頁面出現。最後還需要確認圖片是不是太大。
  • 列印測試：如頁面提供列印功能，需測試是否提供最佳測試模式。

安全測試

  • 目錄設定：Web 安全的第一步就是正確設置目錄。在頁面上選一張圖片，單擊滑鼠右鍵，找到圖片所在的路徑"/objects/images"。然後在瀏覽器地址欄中手工輸入該路徑，發現該站點所有圖片的列表；然後試著進入下一級目錄"/objects" ，在該目錄下有很多資料檔，這樣就是沒做好目錄設定。
  • 登錄：如果網站需要進行用戶登錄，這樣對用戶很方便，可以不需要每次都輸入個人資料。你需要驗證系統阻止非法的帳號登錄，並且能有效驗證。用戶登錄是否有次數限制？如果登錄失敗次數限制是 3，你在第三次登錄的時候輸入正確的帳號密碼，能通過驗證嗎？密碼有規則限制嗎？是否可以不登錄直接瀏覽某個頁面？
  • 腳本語言：每種語言的特性都不同，有些腳本允許訪問根目錄。有經驗的駭客可以將伺服器用戶名跟密碼發送給自己。找出網站使用了哪些腳本語言，並研究該語言的缺陷。

接口測試

測試流程及方法

測試流程

  • 使用測試工具將網站中的錯誤連結找出來；
  • 測試的順序為：ㄧ、由上而下，自左到右；
  • 查看頁面 title 是否正確。 （不只首頁，所有頁面都要查）；
  • LOGO下的一級標題、二級標題的連結是否正確；
  • 首頁登錄、註冊的功能是否完整；
  • 首頁左側欄目下的文章標題、圖片等連結是否正確；
  • 首頁中間欄目下的文章標題、圖片等連結是否正確；
  • 首頁右側欄目下的文章標題、圖片等連結是否正確；
  • 首頁最下方的【友情鏈接】、【關於我們】等連結是否正確；
  • 進入一級選單或二級選單的列表頁。查看左側選單名稱，右側文章列表是否正確；
  • 列表頁的分頁功能是否正確，樣式是否統一；
  • 查看文章詳細頁面的內容是否存在亂碼、頁面樣式是否統一；
  • 站內搜索（各個頁面都要查）功能是否完成；
  • 前後台交互的部分，數據傳遞是否正確；
  • 默認按鈕要支持 Enter 及點擊操作，就是按下 Enter 等同於點擊按鈕。

UI（用戶介面）測試

  • 頁面的樣式風格要統一；
  • 頁面的大小是否一致；同樣的LOGO圖片在各個頁面中顯示是否大小一致；頁面及圖片是否居中顯示；
  • 欄位名稱、文章內容等的文字是否正確，有無錯別字或亂碼；同一級別的字體、大小、顏色是否統一；
  • 提示、警告或錯誤說明應清楚易懂；
  • 切換窗口大小，將窗口縮小後，頁面是否按比例縮小或出現滾動條；各個頁面縮小的風格是否一致，文字是否跨行；
  • 按鈕大小基本相近，最好不要用長的名稱，免得佔用過多的界面位置；按鈕的樣式風格要統一；按鈕之間的間距要一致；
  • 頁面顏色是否統一；前景與背景色搭配是否協調，反差不宜太大，最好少用深色或刺眼的顏色；
  • 若有自動捲動內容或圖片，將滑鼠放在上面，看自動捲動內容或圖片是否停止；
  • 所有的圖片是否都被正確下載，在不同的瀏覽器、分辨率下圖片是否能正確顯示（包括位置、大小）；
  • 文章列表頁，左側的欄目是否與一級、二級欄位的名稱、順序一致；
  • 調整分辨率驗證頁面格式是否有錯位現象；
  • 滑鼠移動到物件焦點上特效是否正確反應，移出焦點特效是否消失；
  • 文字顏色與頁面配色協調，不使用與背景色相近的顏色。
  • 同一界面上的控件最好不要超過10個，多於10個時可以考慮使用分頁顯示。 超過一屏的內容，在底部應有go top按鈕
  • 導航的文字要簡明扼要，字數限制在一行以內
  • 報表顯示時應考慮數據顯示寬度的自適應或自動換行。
  • 所有有數據展現的界面（如統計、查詢、編輯錄入、打印預覽、打印等），必須使測試數據的記錄數超過一屏/一頁，以驗證滿屏/頁時其窗體是否有橫向、縱向滾動條或換頁打(L)印，界面顯示是否正常；
  • 對統計的數據應按用戶習慣進行分類、排序。
  • 用戶在退出系統後重新登錄時應考慮是否需要自動返回到上次退出系統時的位置；
  • 用戶提示應具有一定的指導性，在應用程序正在進行關鍵業務的處理時，應考慮在前台界面提示用戶應用程序正在進行的處理，以及相應的處理過程，在處理結束後再提示用戶處理完畢。
  • 某些數據輸入界面，如果要求輸入的數據符合某項規則，應在輸入界面提供相應的規則描述；當輸入數據不符合規則時應提示用戶是否繼續。
  • 用戶做了任何修改都應該在用戶退出該頁面時提示用戶保存（如果用戶沒有主動保存的情況下）；
  • 在對某些查詢功能進行測試時，應考慮查詢條件的設置的合理性以及查詢結果的互補性。如某些後台處理時間不應該作為查詢條件。
  • 界面測試時，應考慮某一界面上按鈕先後使用的順序問題，以免用戶對此產生迷惑。例如只能在查詢成功後顯示執行按鈕。
  • 在某些對數據進行處理的操作界面，應考慮用戶可能對數據進行處理的頻繁程度和工作量，考慮是否可以進行批量操作。
  • 界面測試不只要考慮合理的鍵盤輸入，還要測試是不是可以用滑鼠複製貼上。
  • 數字精確度顯示的統一：如單價0元，應顯示為0.00元；
  • 時間和日期顯示格式的統一；
  • 對所有可能產生的提示訊息頁面內容和位置進行驗證，確保所有的提示訊息居中。

連結測試

  • 頁面是否有無法連接的內容；圖片是否能正確顯示，是不是有多餘的圖片，頁面是否有空連結（可以用 Google Web Dev Tool 檢查）；
  • 圖片上是不是有沒用的連結；點擊圖片上連結是否跳轉到正確頁面；
  • 點擊首頁或列表頁的文章標題的連結，是否可進入對應的文章頁面；
  • 導航連結的頁面是否正確；是否可依分類連結相應的頁面； （例：【首頁->服務與支持->客戶服務】，分別點擊“首頁”、“服務與支持”、“客服中心”，查看是否可跳轉到正確頁面；）

搜尋測試

  • 搜尋按鈕功能是否實現；
  • 輸入網站中存在的訊息，是否搜尋出正確結果；
  • 輸入鍵盤中所有特殊符號，是否提示錯誤；特別要注意：_ ？ ’ . • \ / -- ；特殊字符
  • 系統是否支持鍵盤後退鍵、Tab鍵；
  • 搜尋結果頁是不是跟其它頁面設計風格一致；
  • 在輸入框輸入空格，點擊搜尋是否彈出錯誤；
  • 站內搜尋輸入框不輸入任何內容，是否搜尋結果為全部或顯示提示信息；
  • 精準查詢還是模糊查詢，如果是模糊查詢輸入：中%國。查詢結果是不是都包含中國兩個字的信息；
  • 焦點放置搜尋框中，搜尋框內容是否被清空；

表單測試

  • 註冊、登錄功能是否正確；
  • 送出、取消按鈕功能是否正確；
  • 修改表單與註冊頁面資料是否一致；
  • 修改表單是否對重複用戶名做驗證；
  • 提交的數據是否能正確保存到後台資料庫（後台資料庫的數據應與前台輸入內容完全一致，數據不會遺失或被改變）；
  • 表單提交，刪除，修改後是否有提示信息；提示、警告、或錯誤說明應該清楚、明暸、恰當。
  • 瀏覽器的前進、後退、刷新按鈕，是否會造成資料重現或頁面報錯；
  • 提交表單是否支持後退鍵和Tab鍵；Tab鍵的順序與控件排列順序要一致，從上到下，從左到右；
  • 下拉列表功能是否正確和資料是否完整（例如：省市下拉列表資料是否互動正確）；

輸入測試

  • 對於手機、郵箱、證件號等的輸入是否有長度及類型的限制；
  • 輸入中文、英文、數字、特殊字符（特別注意單引號和反斜線）及這四類的混合輸入，是否會提示錯誤；
  • 輸入空格、空格+資料 資料+空格，是否提示錯誤；
  • 輸入html語言的 <head>，是否能正確顯示；
  • 輸入全型、半型的英文、數字、特殊符號等，是否提示錯誤；
  • 是否有必填欄位檢查；不輸入必填欄位，是否有友好提示；
  • 輸入超長字段，頁面是否被撐開；
  • 分別輸入大於、等於、小於資料規定限制的資料或數據，是否提示錯誤；
  • 輸入非資料表中的數據類型，是否有友好提示信息；
  • 在文本框中輸入 enter 鍵，顯示頁面是否換行；
  • 錯誤的輸入或操作應有提示說明。

分頁測試

  • 當沒有資料時，首頁、上一頁、下一頁、最後一頁時標籤全部設定成淺灰或不顯示；
  • 在首頁時，“首頁”“上一頁”標籤設定成淺灰；在最後一頁時，“下一頁” “最後一頁”標籤設定成淺灰；在中間頁時，四個標籤均可點擊，且正確跳轉；
  • 翻頁後，列表中的是否資料是否按指定的順序進行排序；
  • 每個分頁標籤是否在同一水平線上；
  • 每個頁面的分頁標籤樣式是否一致；
  • 分頁的總頁數及當前頁數顯示是否正確；
  • 是否能正確跳轉到指定的頁數；
  • 在分頁處輸入非數字的字符（英文、特殊符號等），輸入0或超出總頁數的數字，是否有友好提示訊息；
  • 是否支持 enter 鍵；

數據交互測試

  • 前台的資料操作是否對後台產生正確的影響（如：查看詳細訊息時，需扣除用戶相應的授權）；
  • 可實現前後台資料的互動（如：在線諮詢，能否實現數據的交互即時更新）；數據傳遞是否正確；前後台大數據量信息傳遞數據是否丟失（如500個字符）；多用戶交流時用戶信息控制是否嚴謹；
  • 用戶權限是否隨著授權而變化；
  • 數據未審核時，前台應不顯示；審核通過後，前台應可顯示該條數據；

驗證規則參考

驗證規則 說明
<>0 不等於零，輸入一個不是零的值。
>=0 輸入零或大於零的數字；必須是一個正數。
0 or >100 輸入零或一個大於一百的數。
BETWEEN 0 AND 1 輸入帶百分比符號的值。
<#01/01/2007# 輸入2007年之前的日期
>=#01/01/2007# AND <#01/01/2008# 日期必須在2007年
<Date() 出生日期不能是未來。
StrComp(UCase([LastName]),[LastName],0) = 0 LastName的字段中首字必須為大寫。
>=Int(Now()) 輸入今天的日期。
M Or F 輸入M表示男性，F表示女性。
LIKE "[A-Z]*@[A-Z].com" OR "[A-Z]*@[A-Z].net" OR "[A-Z]*@[A-Z].org" 輸入有效的.com，.net或.org電子郵件地址。
[RequiredDate]<=[OrderDate]+30 輸入日期不可以是30天後的日期。
[EndDate]>=[StartDate] 輸入結束日期在開始日期之前

輸入限制驗證

  • 數據類型：限制用戶輸入時選擇的資料類型。例如：日期/時間格式只接受日期和時間，一個貨幣格式只接受數字，等等。
  • 文字類型：文字類型輸入限制。例如：如> 100 <1000的有效性規則屬性強制用戶輸入100至1000輸入值。規則如[結束日期] >= [起始日期]限制用戶結束日期不可在起始日期之前。或當用戶犯了一個錯誤，如何修復錯誤。
  • 輸入隱碼：使用輸入隱碼強制用戶以特定的方式來輸入以驗證數值。例如，一個輸入隱碼可以強制用戶特定格式輸入日期，例如2007年4月14日。
